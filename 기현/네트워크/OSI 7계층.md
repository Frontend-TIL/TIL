## OSI 7계층

네트워크를 공부하다 보면 가장 먼저 접하는 개념이 OSI 7계층이다.
처음에는 계층 이름만 외우는 느낌이 강한데, 실제로는 **“네트워크 통신을 역할별로 나눠서 이해하기 위한 기준”** 이라고 보면 된다.

---

### OSI 7계층이 왜 필요할까?

OSI 7계층은 네트워크 통신 과정을 한 번에 뭉뚱그려 보는 게 아니라, **각 단계의 역할을 분리해서 이해하고 관리하기 위해** 필요하다.

내가 이해한 핵심 이유는 크게 3가지다.

**1. 역할이 나뉘어서 관리하기 쉽다**

각 계층이 담당하는 역할이 다르기 때문에, 한 부분이 바뀌어도 다른 부분까지 전부 수정할 필요가 없다.
즉, 독립성과 모듈화에 도움이 된다.

**2. 문제가 생겼을 때 원인 추적이 쉽다**

네트워크 장애가 발생했을 때
“케이블 문제인지”, “IP 문제인지”, “포트 문제인지”, “애플리케이션 문제인지”를
계층별로 나눠서 볼 수 있다.

**3. 표준화에 유리하다**

서로 다른 회사가 만든 장비나 소프트웨어라도, 같은 계층 구조와 프로토콜 규칙을 따르면 통신할 수 있다.
즉, 상호 호환성을 높이는 데 도움이 된다.

---

### OSI 7계층과 TCP/IP 4계층의 차이

OSI 7계층은 개념적으로 통신을 설명하기 위한 모델에 가깝고,
TCP/IP 4계층은 실제 인터넷에서 사용되는 실무 중심 모델에 가깝다.

쉽게 말하면,

**OSI 7계층**: 네트워크를 배우고 구조를 이해하기 좋은 모델
**TCP/IP 4계층**: 실제 통신 환경에 맞게 단순화해서 사용하는 모델

OSI는 계층을 더 세분화해서 나누고,
TCP/IP는 실제 구현 관점에서 몇몇 계층을 합쳐서 사용한다.

| 구분   | OSI 7계층     | TCP/IP 4계층    |
| ---- | ----------- | ------------- |
| 성격   | 이론적, 개념적 모델 | 실무적, 구현 중심 모델 |
| 계층 수 | 7계층         | 4계층           |
| 특징   | 역할 구분이 세밀함  | 실제 사용에 맞게 통합됨 |


**계층 통합 관점**
- OSI의 세션/표현/응용 계층(5~7계층) → TCP/IP에서는 Application 계층으로 묶임
- OSI의 물리/데이터링크 계층(1~2계층) → TCP/IP에서는 Network Access 계층으로 묶임

---

### 패킷 손실은 어느 계층 문제일까?

패킷 손실은 보통 **네트워크 계층(3계층)** 에서 많이 언급된다.
특히 라우터에서 패킷을 전달하는 과정에서 혼잡이 생기면 버퍼가 가득 차고, 그때 패킷이 버려질 수 있다.

내가 이해한 흐름은 이렇다.

라우터는 들어온 패킷을 잠깐 **버퍼**에 저장한 뒤 다음 라우터나 목적지로 보낸다.
그런데 패킷이 들어오는 속도가 나가는 속도보다 더 빠르면 큐에 쌓이게 되고,
이게 계속 심해지면 버퍼 용량을 넘어서면서 **패킷 손실(loss)** 이 발생한다.

즉,

- 혼잡 → 큐 지연 증가
- 버퍼 초과 → 패킷 손실 발생

이 과정은 주로 **3계층 장비인 라우터**에서 발생한다고 볼 수 있다.

다만 손실된 패킷을 **재전송해서 복구하는 책임**은 **보통 전송 계층(4계층, TCP)** 이 맡는다.
그래서 “손실은 어디서 발생하냐”와 “복구는 누가 하냐”를 구분해서 생각해야 한다.

---

### 네트워크 지연은 왜 생길까?

패킷 손실이랑 같이 자주 나오는 개념이 지연(delay)이다.
지연은 보통 아래 4가지로 나눠서 본다.

**1. 처리 지연**

라우터가 패킷 헤더를 읽고, 목적지를 확인하고, 오류를 검사하는 데 걸리는 시간이다.

**2. 큐 지연**

패킷이 바로 전송되지 못하고 버퍼에서 기다리는 시간이다.
이건 네트워크 혼잡도에 따라 달라져서 예측이 어렵다.

**3. 전송 지연**

패킷의 비트들을 링크 위로 실제 밀어내는 데 걸리는 시간이다.
대역폭이 낮을수록 더 오래 걸린다.

- 전송 지연 = 패킷 길이 / 대역폭

**4. 전파 지연**

신호가 실제 매체를 따라 이동하는 데 걸리는 시간이다.
거리와 매체 종류의 영향을 받는다.

- 전파 지연 = 거리 / 전파 속도

결국 네트워크 성능은 단순히 빠른 장비만으로 결정되지 않고,
**혼잡도, 대역폭, 거리**가 같이 영향을 준다.

---

### 웹사이트 접속이 안 될 때 OSI 7계층으로 어떻게 볼까?

이런 질문은 면접에서도 자주 나오는 것 같다.
나는 하위 계층부터 상위 계층으로 올라가며 확인한다고 답하면 자연스럽다고 생각한다.

**1. 물리 계층**
- 케이블이 제대로 연결되어 있는지
- 공유기, 허브, 스위치 전원이 켜져 있는지
  
**2. 데이터링크 계층**
- 같은 LAN 안에서 통신이 되는지
- MAC 레벨에서 문제가 없는지

**3. 네트워크 계층**
- IP를 정상적으로 받았는지
- 게이트웨이나 라우팅에 문제가 없는지
- 외부 인터넷과 통신이 되는지

**4. 전송 계층**
- 서버의 특정 포트가 열려 있는지
- 방화벽이 포트를 막고 있지는 않은지

**5. 세션/표현 계층**
- TLS/SSL 인증서 문제는 없는지
- 세션 유지 과정에서 오류가 없는지

**6. 응용 계층**
- 웹 서버(Nginx, Apache 등)가 정상 동작하는지
- DNS 설정이 올바른지
- 애플리케이션 자체에서 에러가 난 건 아닌지

**DNS는 어느 계층일까?**

DNS는 IP 주소를 찾는 역할 때문에 헷갈리기 쉬운데,
**DNS 자체는 애플리케이션 계층(7계층) 프로토콜**이다.

---

### 클라이언트에서 서버로 데이터가 갈 때 어떻게 처리될까?

클라이언트가 서버로 데이터를 보낼 때는 캡슐화,
서버가 받을 때는 역캡슐화가 일어난다.

즉, 위에서 만든 데이터가 아래 계층으로 내려가면서 필요한 정보가 계속 붙고,
받는 쪽에서는 그걸 다시 하나씩 벗겨낸다.

---

### 캡슐화 과정

**1. 애플리케이션 계층**

사용자가 보낸 HTTP 요청 같은 실제 데이터가 만들어진다.

**2. 전송 계층**

여기에 TCP 헤더가 붙으면서 세그먼트가 된다.
이때 포트 번호 같은 정보가 포함된다.

**3. 네트워크 계층**

IP 헤더가 붙으면서 패킷이 된다.
출발지/목적지 IP 주소가 들어간다.

**4. 데이터링크 계층**

MAC 주소 정보와 오류 검출용 트레일러가 붙으면서 프레임이 된다.

**5. 물리 계층**

최종적으로 0과 1의 비트 신호로 바뀌어 실제 매체를 통해 전송된다.

---

### 역캡슐화 과정

서버는 받은 데이터를 아래에서 위로 다시 해석한다.

**1. 물리 계층**

전기/광 신호를 비트로 바꾼다.

**2. 데이터링크 계층**

MAC 주소를 확인하고 프레임 정보를 처리한다.

**3. 네트워크 계층**

IP 주소를 보고 자신의 패킷인지 확인한다.

**4. 전송 계층**

포트 번호를 보고 어떤 프로세스에 전달할지 결정한다.

**5. 애플리케이션 계층**

최종적으로 HTTP 요청 같은 원래 데이터가 애플리케이션에 전달된다.

---

### 각 계층별 데이터 단위(PDU)

데이터는 계층을 지날 때마다 부르는 이름이 조금씩 달라진다.
이걸 **PDU(Protocol Data Unit)** 라고 한다.

**1. 응용 / 표현 / 세션 계층**
- 데이터(Data) 또는 메시지
- 사용자가 실제로 주고받는 내용

**2. 전송 계층**
- 세그먼트(TCP)
- 데이터그램(UDP)

**3. 네트워크 계층**
- 패킷(Packet)

**4. 데이터링크 계층**
- 프레임(Frame)

**5. 물리 계층**
- 비트(Bit)

> 프론트엔드에서 OSI 7계층은 단순한 이론이 아니라,
> 네트워크 문제를 계층별로 나누어 원인을 빠르게 파악하고
> 백엔드 및 인프라와 협업하기 위한 기준이 된다.
