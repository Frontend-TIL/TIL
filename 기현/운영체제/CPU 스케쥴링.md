### CPU 스케줄링

---

- **CPU Scheduling이란 무엇인가요?**
    
    CPU를 효율적으로  사용하기 위해 어떤 프로세스를 먼저 실행할 지 결정하는 정책입니다.
    
    - 여러 프로세스 중 **누가 CPU를 사용할지 결정하는 정책**
    - 목적:
        - CPU 놀지 않게 (사용률 ↑)
        - 빠르게 응답 (Response ↓)
        - 공정하게 분배 (기아 ↓)

- **선점형과 비선점형의 차이는 무엇인가요?**
    
    선점형은 OS가 CPU를 강제로 회수할 수 있고, 비선점형은 프로세스가 끝날 때까지 CPU를 사용합니다.
    
    - **선점형**
        - OS가 강제로 CPU 뺏음
        - 예: Round Robin
        - 응답 빠름 / 오버헤드 있음
    - **비선점형**
        - 끝날 때까지 보장
        - 예: FCFS, SJF
        - 예측 쉬움 / 응답 느림

- **FCFS, SJF, Round Robin 알고리즘을 설명해주세요.**
    
    FCFS는 도착 순서, SJF는 실행 시간 기준, Round Robin은 일정 시간씩 나눠 실행합니다.
    
    대표 알고리즘 3가지
    
    - **FCFS**: 먼저 온 순서
    - **SJF**: 짧은 작업 먼저
    - **Round Robin**: 시간 쪼개서 번갈아 실행

- **SJF가 평균 대기 시간을 줄일 수 있는 이유는 무엇인가요?**
    
    SJF는 짧은 작업을 먼저 하는데, 이때 짧은 것을 먼저 처리하면 전체 대기 시간이 줄어 평균 대기 시간이 최소화됩니다.
    

- **SJF는 왜 실제 시스템에서 잘 사용되지 않나요?**
    
    실행 시간을 미리 알기 어렵고, 긴 작업이 계속 뒤로 밀리는 Starvation 문제가 발생합니다.
    

- **Round Robin에서 Time Quantum이 중요한 이유는 무엇인가요?**
    
    → Time Quantum이 너무 크면 프로세스가 CPU를 오래 점유하게 되어 선점이 거의 발생하지 않으므로 FCFS처럼 동작합니다.
    
    → 반대로 너무 작으면 프로세스 간 전환이 자주 발생해 Context Switching 비용이 증가하여 CPU 효율이 떨어집니다.
    
    ### Time Quantum = CPU를 얼마나 자주 나눌 것인가….
    
    - 크면 → FCFS처럼 됨 == 한 번 들어오면 끝까지 실행한다.
    - 작으면 → 컨텍스트 스위칭 많아짐 (비효율) == 계속 바뀌느라 오버헤드 증가
    
    Context Switching = CPU가 다른 프로세스로 바뀔 때 상태 저장 및 복구하는 것.
    

- **실제 시스템에서는 어떤 스케줄링 방식이 사용되며, 그 이유는 무엇인가요?**
    
    실제 OS = 여러 알고리즘 섞어서 쓴다 → MLFQ
    
    실제 시스템에서는 단일 알고리즘이 아니라 Multilevel Feedback Queue 기반의 스케줄링을 사용합니다.
    → 이는 짧은 작업은 빠르게 처리하고, 긴 작업도 공정하게 실행되도록 하여 응답성과 효율성을 동시에 확보하기 때문입니다.
